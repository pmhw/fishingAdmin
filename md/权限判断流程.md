# 后台权限判断流程

## 一、整体顺序

```
请求进入 → 路由匹配 → AdminAuth 中间件 → AdminPermission 中间件 → 控制器
                ↓                    ↓
            未登录 401            无权限 403
                ↓                    ↓
            写入 adminId          放行 → 控制器内（部分接口再做「数据范围」校验）
```

---

## 二、第一步：AdminAuth（登录校验）

**位置**：`app/middleware/AdminAuth.php`  
**作用**：只做「是否登录」校验，不判断具体权限。

1. 从请求头取 `Authorization`，去掉 `Bearer ` 得到 token。
2. 用 token 调 `Auth::getAdminIdByToken($token)`（查缓存里是否还有该 token 对应的管理员 id）。
3. 若查不到或无效 → 返回 **401**，提示「未登录或登录已过期」。
4. 若有效 → 把管理员 id 写入 **`$request->adminId`**，交给下一个中间件。

**结果**：通过则请求上一定带有 `adminId`，后续逻辑都用它来查「当前登录人」和其权限。

---

## 三、第二步：AdminPermission（权限码校验）

**位置**：`app/middleware\AdminPermission.php`  
**作用**：按「请求方法 + 路径」查出需要的权限码，再判断当前管理员是否拥有该权限码。

### 3.1 路径归一化

- 从 `$request->pathinfo()` 去掉前缀 `api/`、`admin/`，得到纯路径，例如：  
  `admin-users`、`roles/3`、`configs/values`、`ponds/1`。

### 3.2 是否跳过权限校验

- 若路径是 **`me`** 或 **`logout`**（或以其为前缀），则**不查权限**，直接放行（只要求已登录）。

### 3.3 查「本接口需要什么权限」

- 配置在 **`config/admin_permission.php`**：键为 `METHOD:path`，值为权限码。
- 例如：`GET:ponds` → `admin.pond.manage`，`PUT:ponds/` → `admin.pond.manage`（`path/` 可匹配 `ponds/1` 等）。
- 用当前请求的 **方法 + 归一化路径** 去匹配：
  - 先精确匹配 `METHOD:path`；
  - 再匹配「以 `METHOD:path/` 开头」的规则（用于带 id 的路径）。
- 若**没有匹配到任何规则** → 认为本接口**不要求权限码**，直接放行（仅需登录）。

### 3.4 查「当前管理员有哪些权限」

- 用 `$request->adminId` 查 **AdminUser**，再调 **`$user->getPermissionCodes()`**：
  - 该管理员关联一个 **AdminRole**（角色）；
  - 角色若为 **超级管理员**（code = `super_admin`）→ 返回 **`['*']`**（视为拥有所有权限）；
  - 否则从 **admin_role_permission** 关联的 **AdminPermission** 里取所有 **code**，返回一维数组，如 `['admin.venue.manage', 'admin.pond.manage']`。

### 3.5 是否放行

- 若当前管理员权限列表中有 **`*`** → 放行。
- 若当前管理员权限列表中包含 **本接口所需权限码** → 放行。
- 否则 → 返回 **403**，提示「无权限访问」。

**小结**：  
- 未配置在 `admin_permission.php` 的接口：只校验登录，不校验权限码。  
- 已配置的接口：必须拥有对应权限码（或超级管理员）才能访问。

---

## 四、第三步：控制器内「数据范围」校验（仅部分接口）

通过 AdminAuth + AdminPermission 后，只说明「有权限访问这个接口」。  
部分接口还会在**控制器内部**再做一层「数据范围」限制，目前只有**池塘相关**做了这类逻辑。

### 4.1 池塘相关（PondScopeTrait）

**涉及**：`PondController`、`PondRegionController` 中列表、详情、增删改等。

**逻辑**（在 `PondScopeTrait::getAdminAllowedPondIds()` 等）：

1. 用 `$request->adminId` 取当前 **AdminUser** 及其 **AdminRole**。
2. 若该角色权限里**没有** `admin.pond.manage` → 视为**无池塘权限**，列表为空、其它操作 403。
3. 若角色是**超级管理员**（或等价有 `*`）→ **允许管理全部池塘**（不查表）。
4. 若角色有 `admin.pond.manage`：
   - 查 **admin_role_pond** 表：该 **role_id** 是否绑定了具体 **pond_id**；
   - **没有记录** → 允许管理**全部池塘**；
   - **有记录** → 只允许管理表中列出的那些 **pond_id**。

**在接口中的使用**：

- **列表**：只查「允许的 pond_id」范围内的数据；并返回 `can_delete`（仅「全部池塘」时可删除）。
- **详情/编辑/删除**：先判断该条记录的池塘 id 是否在「允许的 pond_id」里，否则 403。
- **新增**：仅当「允许全部池塘」时才允许新增，否则 403。

所以：**权限码**决定能不能进「池塘管理」接口；**数据范围**决定能看/改/删哪些池塘、能不能新增。

## 五、流程简图

```
[请求] GET /api/admin/ponds/1
    ↓
[AdminAuth] 校验 Token → 写入 adminId
    ↓
[AdminPermission]
    路径 → "ponds/1"
    配置 → GET:ponds/ 需要 "admin.pond.manage"
    当前用户权限 → AdminUser(adminId) → Role → getPermissionCodes()
    有 "admin.pond.manage" 或 "*" ? → 是：放行
    ↓
[PondController::detail(1)]
    getAdminAllowedPondIds() → 该角色能管哪些池塘
    canAccessPond(1) ? → 是：返回详情；否：403
```

## 六、配置与数据对应关系

| 层级 | 配置/数据 | 作用 |
|------|-----------|------|
| 路由 | `route/app.php` | 决定哪些 URL 走 AdminAuth + AdminPermission。 |
| 权限映射 | `config/admin_permission.php` | 决定「某方法+路径」需要哪个**权限码**。 |
| 权限定义 | 表 `admin_permission` | 权限码（如 admin.pond.manage）的定义。 |
| 角色权限 | 表 `admin_role_permission` | 角色拥有哪些权限（多对多）。 |
| 管理员角色 | `admin_user.role_id` | 管理员属于哪个角色。 |
| 池塘范围 | 表 `admin_role_pond` | 某角色只能管哪些池塘（为空=全部）。 |

新增一个「需要权限」的接口时：  
1. 在 `admin_permission` 表（或后台「角色与权限」）里加权限码；  
2. 在 `config/admin_permission.php` 里为该接口的 **METHOD:path** 配置该权限码；  
3. 需要「按池塘细分」时，在控制器里使用 `PondScopeTrait` 或类似逻辑做数据范围校验。
