# 后台管理权限设计（RBAC）

## 一、模型说明

- **管理员** `admin_user`：归属一个**角色**（role_id）。
- **角色** `admin_role`：如「超级管理员」「运营」「客服」。
- **权限** `admin_permission`：最小控制粒度，用**权限码**表示，如 `admin.user.list`、`admin.banner.edit`。
- **角色-权限**：多对多，一个角色拥有多个权限；一个权限可属于多个角色。

这样：**管理员 → 角色 → 权限**，判断是否有权做某操作 = 判断该管理员所属角色是否拥有对应权限码。

---

## 二、表结构（在现有 admin_user 基础上扩展）

### 1. 角色表 admin_role

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| name | varchar(50) | 角色名称，如「超级管理员」 |
| code | varchar(50) | 角色编码，如 super_admin、operator |
| description | varchar(255) | 备注 |
| created_at / updated_at | datetime | 时间 |

### 2. 权限表 admin_permission

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| name | varchar(50) | 展示名称，如「管理员列表」 |
| code | varchar(80) | 权限码，唯一，如 admin.user.list、admin.banner.manage |
| module | varchar(50) | 模块/分组，如 user、banner，便于前端分组展示 |
| created_at / updated_at | datetime | 时间 |

### 3. 角色-权限关联表 admin_role_permission

| 字段 | 类型 | 说明 |
|------|------|------|
| role_id | int | 角色 id |
| permission_id | int | 权限 id |
| 主键 | (role_id, permission_id) | 联合主键 |

### 4. admin_user 表增加字段

- **role_id**（int，可空）：所属角色 id。为 null 时可约定为「未分配角色」或沿用你现有逻辑（如默认超级管理员）。

---

## 三、权限码约定（示例）

建议按「模块.资源.操作」命名，方便前后端统一：

| 权限码 | 说明 |
|--------|------|
| admin.user.list | 管理员列表查看 |
| admin.user.create | 新增管理员 |
| admin.user.update | 编辑管理员 |
| admin.user.delete | 删除管理员 |
| admin.banner.list | 轮播图列表 |
| admin.banner.edit | 轮播图编辑 |
| admin.role.manage | 角色与权限配置（仅超管） |

新增业务时，在 `admin_permission` 里加一条记录即可，再在角色里勾选该权限。

---

## 四、后端校验流程

1. **登录 / 获取当前用户信息**  
   除返回用户基本信息外，根据 `admin_user.role_id` 查 `admin_role_permission` + `admin_permission`，得到该用户拥有的**权限码列表**（如 `['admin.user.list','admin.user.create']`），一并返回给前端。

2. **接口鉴权**  
   - 需要「登录即可访问」的接口：只走现有 **AdminAuth** 中间件（校验 token，写入 adminId）。  
   - 需要「特定权限」的接口：在 AdminAuth 之后再做**权限校验**：  
     - 从请求中取出当前 adminId，查出其权限码列表；  
     - 判断当前接口要求的权限码是否在该列表中（或是否超级管理员默认拥有全部权限）；  
     - 有则 `$next($request)`，无则返回 403。

3. **与路由对应**  
   每个需要权限控制的路由，对应一个权限码（如 GET /api/admin/admin-users 对应 `admin.user.list`）。可在路由上挂「权限码」参数，或在控制器里写死校验逻辑。

---

## 五、前端使用方式

1. **菜单**  
   根据接口返回的权限码列表，只展示有权限的菜单项（如没有 `admin.user.list` 则不显示「管理员管理」或对应菜单）。

2. **按钮**  
   如「新增」「删除」根据 `admin.user.create`、`admin.user.delete` 是否存在权限列表中来控制显示或禁用。

3. **路由**  
   若前端做了路由级权限，进入某页前可判断是否有对应权限码，无则跳转 403 或首页。

---

## 六、超级管理员约定

- **超级管理员**：可约定为 `admin_role.code = 'super_admin'`，或 `admin_user.role_id` 为某个固定 id。  
- 校验时：若当前用户是超级管理员，则视为拥有所有权限，不再查 `admin_role_permission`。  
- 新建的默认 admin 账号可设为「超级管理员」角色。

---

## 七、实施顺序建议

1. 建表：`admin_role`、`admin_permission`、`admin_role_permission`，并给 `admin_user` 增加 `role_id`。  
2. 初始化数据：插入超级管理员角色、常用权限码、给默认 admin 赋超级管理员角色。  
3. 后端：登录/me 接口返回 permissions；新增「权限校验」中间件或在控制器中按权限码校验。  
4. 前端：根据 permissions 控制菜单和按钮。
