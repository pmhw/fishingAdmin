# 钓场店铺 · 公共商品库设计说明

## 需求对应

| 需求 | 设计 |
|------|------|
| 公共商品库 | `product`（SPU）+ `product_sku`（规格） |
| 某个店拥有商品 | `venue_product`：钓场 + SPU 上架关系 |
| 设置库存 | `venue_product_sku.stock`（按店、按规格） |
| 不同规格、不同价格 | `product_sku` 定义规格；`venue_product_sku.price` 为**店内实际售价**（可每店不同） |

## 实体关系（文字 ER）

```
product (公共 SPU)
   └── product_sku (多条规格，可选 default_price 作建议价)

fishing_venue (已有)
   └── venue_product (本店上架了哪些 SPU)
            └── venue_product_sku (本店每个 SKU：price + stock + 是否可售)
```

- **不在 `product` 上写售价**：避免「公共库一个价」与「各店不同价」冲突。  
- **店内唯一售价来源**：`venue_product_sku.price`。  
- **`product_sku.default_price`**：运营维护的建议价，后台「添加到本店」时可自动带出，允许店员再改。

## 典型流程

1. **运营维护库**  
   - 新建 `product` → 添加多条 `product_sku`（规格名 + 可选 `spec_json` + 建议价）。

2. **某钓场上架**  
   - 插入 `venue_product(venue_id, product_id)`。  
   - 对该 SPU 下要卖的每条 `product_sku` 插入 `venue_product_sku`：填 `price`、`stock`。

3. **小程序展示**  
   - 列表：`venue_product` JOIN `product`，`status=1`。  
   - 详情/下单：`venue_product_sku` JOIN `product_sku`，`venue_product_sku.status=1`，展示 `price`、`stock`。

4. **库存**  
   - 下单成功扣减对应 `venue_product_sku.stock`（建议配合订单明细表，另做）。

## 扩展建议（未建表）

- **订单**：`shop_order` + `shop_order_item`（含 `venue_product_sku_id`、单价、数量）。  
- **称重商品**：`stock` 改为 `DECIMAL(10,3)`。  
- **分类**：已预留 `product_category`，可按需启用。

## SQL 脚本

执行：`md/venue_shop_product.sql`（注意库中需已存在 `fishing_venue`，且引擎为 InnoDB 以使用外键）。

若环境不便使用外键，可去掉 `CONSTRAINT` 段落，在应用层保证引用完整性。
